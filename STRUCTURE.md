# Repo structure

A one-screen map of the repo. Updated 2026-05-15 after the
"`openclaw-skills/` → `plugin/`" cleanup pass.

## The top-level

```
/
├── apps/                 ← Deployable Next.js apps (Coolify services)
│   ├── dashboard/        ← The verifier UI at agentscan.online / mainnet.agentscan.online
│   └── docs/             ← The end-user docs site at docs.agentscan.online
│
├── plugin/               ← THE OpenClaw plugin (we ship exactly one).
│   ├── src/              ← TypeScript source
│   ├── tests/            ← vitest suites
│   ├── dist/             ← esbuild bundle output (gitignored, npm-publish target)
│   ├── openclaw.plugin.json   ← OpenClaw manifest (config schema)
│   ├── build.mjs         ← Bundler — produces a self-contained dist/ for npm publish
│   └── package.json      ← Workspace package `@verifiable-agent-execution/plugin`
│                            (npm-publish name is different: `@blockchainoracle/openclaw-verifiable-execution`)
│
├── packages/             ← Shared workspace libraries (imported by apps/ + plugin/)
│   ├── chain-client/     ← AgenticIDClient + SessionAnchor (on-chain reads + mints)
│   ├── logger/           ← SessionLogger + 0G Storage upload
│   └── tee-adapter/      ← TEE signing helpers + MockTEEVerifier ABI
│
├── contracts/            ← Solidity (Hardhat). AgenticID.sol + MockTEEVerifier.sol.
│
├── scripts/              ← Utility scripts run via `pnpm tsx`
│   ├── smoke/            ← 3 re-runnable live-testnet smoke tests
│   ├── init-wallet.ts    ← Plugin first-run wallet generator
│   └── capture-anchor.ts ← Playwright anchor-screenshot capture
│
├── tests/visual/         ← Playwright visual regression for the dashboard.
│                            (Lives at root because playwright.config.ts is at root;
│                            this is the only spec file here.)
│
├── nixpacks.toml         ← Multi-target Coolify build (APP_NAME picks dashboard or docs)
├── install.sh            ← One-shot installer for cloning + linking the plugin locally
├── README.md             ← User-facing entrypoint (links to docs.agentscan.online)
├── CLAUDE.md             ← AI-agent harness instructions (gitignored on public repo)
└── pnpm-workspace.yaml   ← Workspace globs: apps/* packages/* plugin contracts
```

## Why these names

- **`apps/`** and **`packages/`** are standard pnpm-workspace
  convention. `apps/` are deployables (each gets its own Coolify
  service). `packages/` are libraries imported by apps and the
  plugin. Don't read more into the split than that.

- **`plugin/`** is singular and unscoped on purpose. We only ever
  ship ONE OpenClaw plugin from this repo. Calling the folder
  `plugin/` (not `plugins/verifiable-execution/`) reflects that.
  The npm-publish name still includes the `verifiable-execution`
  string for backwards compat and discoverability.

- **`plugin/dist/`** is where the esbuild bundle lands. It lives
  inside `plugin/` (not at a separate top-level `dist-plugin/`)
  because the dist is part of the plugin's lifecycle. Gitignored;
  regenerated by `pnpm --filter @verifiable-agent-execution/plugin build`.

- **`contracts/`** is the conventional name for Hardhat projects.
  Kept at root because Hardhat looks for `contracts/` and
  `hardhat.config.ts` together.

- **`tests/visual/`** at root rather than under `apps/dashboard/tests/`
  because `playwright.config.ts` lives at root (alongside `tsconfig.json`
  and `pnpm-workspace.yaml`). Moving the spec dir would split the
  Playwright config away from its spec.

## What was renamed (2026-05-15)

| Old name | New name | Why |
|---|---|---|
| `openclaw-skills/verifiable-execution/` | `plugin/` | Only one plugin, ever. OpenClaw renamed "skills" → "plugins" in 2026.4. |
| `dist-plugin/verifiable-execution/` | `plugin/dist/` | Dist artifact belongs next to its source, not at repo root. |
| `@verifiable-agent-execution/openclaw-skill` (workspace pkg) | `@verifiable-agent-execution/plugin` | Matches the new folder name. |

The npm-publish name (`@blockchainoracle/openclaw-verifiable-execution`)
did NOT change — that's the user-facing identifier and renaming it
would break every existing install.
