## 1. Why this pivot (the framing problem)

Abu flagged that v0.1–0.2 builds a **public Etherscan-style log of agent activity**. Two product issues:

1. **Privacy.** Real users will never share a URL of their AI prompts/responses with the world. Public-by-default = product-wrong.
2. **Fragmentation.** Per-turn iNFT minting with sparse content makes real usage look worse than the scripted demo. The demo (tokenId 0) constructed 4 entries deliberately; real Telegram tokens have 1–2 thin entries.

Three parallel research agents (saved transcripts at `/private/tmp/.../tasks/`) converged on the same answer:

- **Every mature project** (Phala dstack, Ritual, Sentient, Bittensor, Theoriq, Olas, EAS, NANDA, 0G AgentCourt) encrypts content. On-chain = hashes + attestations only. Plaintext lives encrypted off-chain.
- **0G's design intent** for iNFTs is "tokenized agent identity with encrypted metadata," not "public transaction log." Canonical examples: mint-and-manage, authorize-and-delegate, marketplace-trading. We're ~30% aligned today.
- **EAS PrivateData pattern** is the textbook selective-reveal primitive: Merkle root anchored on-chain, multi-proof for partial reveal, verified client-side.

**v0.3.0 pivot in one sentence**: *encrypted-by-default agent receipts, anchored on 0G, selectively revealable via owner-generated share links.*

## 3. Architecture (data flow)

### Plugin write-path (agent_end fires) — SILENT

```
1. SessionLogger has buffered entries (already signed per v0.2.0)
2. Plugin generates random 32-byte symmetric key K and 12-byte IV
3. plaintext = JSON.stringify(SessionLog)
4. ciphertext + tag = AES-256-GCM-encrypt(plaintext, K, IV)
5. envelope = {v:1, alg:"aes-256-gcm", iv: hex(IV), ciphertext: hex(c), tag: hex(t)}
6. upload(buffer = JSON.stringify(envelope)) → 0G Storage → rootHash
7. iMint(agentId, [{dataDescription: "exec-log:<sessionId>:<modelId>", dataHash: rootHash}]) → tokenId
8. keystore.put(tokenId, K) ← persist locally, mode 0600
9. Update "last-receipt" pointer (keystore writes a small file recording the latest tokenId for quick /share lookup)
10. structuredLog INFO — visible in gateway log for operator grep, but NOT injected into user-visible chat
11. NO URL pollution in any user-visible channel by default
```

### Plugin share-path (user types `/share` in Telegram)

```
1. OpenClaw routes inbound message to inbound_claim handlers
2. Plugin detects /share command match (via registerCommand at register time)
3. Parse: `/share` → most recent tokenId; `/share 7` → token 7
4. Look up key in keystore by tokenId
5. Build shareLink = `${verifyUrlBase}/verify/${tokenId}#k=${base64url(K)}`
6. Return {handled: true, reply: {text: "📎 Receipt for tokenId <id>:\n\n<URL>\n\nAnyone with this link can verify the proof chain + see decoded entries. Without the link's `#k=...` fragment, viewers see only metadata."}}
7. User sees the URL ONLY when they asked for it
```

### Dashboard read-path (as-shipped — KEY-BLIND server per Codex round-1)

```
GET /verify/7                                  ← cold visitor
  ↓ server-side: resolveProof("7") — NO key parameter (server is key-blind)
  ↓ getIntelligentDatas + indexer.downloadToBlob(rootHash)
  ↓ detect envelope → return {verified:"encrypted", entries OMITTED, meta:{rootHash, sessionId, rpcUrl, verifierAddress, …}}
  ↓ client-side: EncryptedReveal reads window.location.hash for #k=…
  ↓ if absent: render "🔒 Encrypted — request reveal link from owner"
  ↓ if present:
  ↓   fetch("/api/verify/7/blob") ← NEW route; key-blind passthrough returns the envelope JSON
  ↓   WebCrypto.subtle.decrypt(envelope, K)    ← decrypts in the browser, key NEVER hits server
  ↓   synthesize ProofResponse with decoded entries
  ↓   render SessionView with client-side verifyEntry callback
  ↓     (browser ethers does ecrecover + on-chain MockTEEVerifier.verifyTEESignature)
  ↓ either way: hash/sig/anchor badges flip green (those work without plaintext)
```

### Critical ordering for crash recovery

```
Per agent_end:
  generate K
  keystore.put(tempId = sessionKey, K)         ← BEFORE flush; if crash here, key still on disk
  upload ciphertext → rootHash
  iMint → tokenId
  keystore.rename(tempId → tokenId)             ← AFTER mint; permanent association
  emit shareLink
```

If we crash between `flush` and `mint`, the operator has the rootHash + key + entryCount and can `retryMint` (existing recovery primitive in `SessionAnchorMintAfterFlushError`).

## 4. Demo asset design

**Three demo URLs the submission shows judges:**

1. **`/verify/0`** (full-decode-baked-in via `DEMO_KEY`) — the legacy DeFi swap demo. Click cold → 4 entries, all green badges, TEE Verified (signed by 0G Compute's TDX-attested provider). **Shows: what a maximum-fidelity public proof looks like.**

3. **`/verify/<new-v030-token-no-key>`** — same token, without the key. **Shows: someone with the URL but no key still verifies the proof exists; cannot read content.**

## 7. Risks & mitigations (from research agent 2)

| Risk | Mitigation |
|---|---|
| Fragment isn't server-visible → 2-pass render risk | Page.tsx becomes a server-renders-shell + client component hydrates with `window.location.hash`. Pattern documented in Next 14 App Router examples. |
| `sessionLogSchema.parse` rejects envelopes | Branch on first-byte detection (envelope starts with `{"v":1,...` JSON; plaintext starts with `{"sessionId":...`) — both are well-formed JSON, distinguishable by required fields. |
| Token 0 (legacy plaintext) breaks if dashboard always expects envelopes | Detection branch handles both. Backward-compat by design. |
| Operator on host A can't share-link from host B | Document: keystore.json is portable + must travel with operator's wallet. Backup alongside wallet. |
| Crash between flush + keystore.put | Critical ordering above — keystore.put goes FIRST under temp name. |
| §14 grep gate trips on "mock"/"dummy" in crypto comments | Use `testKey`, `fixedIv`, `validatorKey`, etc. |
| Playwright visual baselines break | Anchor screenshots are immutable; baseline screenshots update with `--update-snapshots`. Encrypted state IS the new baseline for `/verify/<new-token>`; legacy `/verify/0` baseline preserved. |

## 8. Open questions for Abu before code

1. **EAS PrivateData Merkle tree for field-level partial reveal?** Strictly nicer than all-or-nothing per-token reveal. Adds ~1 day. Recommendation: defer to v0.4.0 — v0.3.0 ships token-level reveal which is already a huge UX improvement.

2. **CLI subcommand `openclaw verifiable-execution share-link <tokenId>`?** Useful for operators who want to share a specific token retroactively. Adds ~30 min. Recommendation: include — it's small and earns demo points.

3. ~~**`before_message_write` hook to inject share link into agent's Telegram reply?**~~ **REJECTED by Abu 2026-05-12.** Auto-appending a URL to every agent reply is FRICTION, not magic — most agent uses don't need sharing, and polluting every reply with "🔒 Verify: …" looks AI-developer-y and breaks the conversation flow. Replaced with:

   **NEW: `registerCommand("share")` + `inbound_claim` hook for slash-command-driven share.**
   - User types `/share` in Telegram → bot's command menu already lists it (OpenClaw auto-registers with channel providers)
   - OpenClaw routes the inbound to our plugin via `inbound_claim` (only when `commandAuthorized: true` for our `/share` command)
   - Plugin handler looks up the most recent token + key from keystore, returns `{handled: true, reply: {text: "Receipt: https://verifiable.0g.ai/verify/<id>#k=<key>"}}`
   - `/share <tokenId>` syntax for specific token
   - **Default reply behavior: silent.** No URL pollution. URL surfaces ONLY when the user asks.
   - **Works everywhere**: Telegram, Discord, CLI, any channel — `registerCommand` is channel-agnostic.
   - Reference impl: stock `codex` plugin at `/tmp/openclaw-src/extensions/codex/index.ts:32-38` uses exactly this pattern.
   - Adds ~1 hour (vs the auto-inject's 45 min — more code, but RIGHT code).

4. **Update demo materials (DEMO.md, README, submission deck)?** v0.3.0 changes the pitch. Adds ~2 hours. Recommendation: include — submission deadline is May 16, this is essential.

5. **Add `plaintextHash` field to iNFT metadata?** Per research agent 2, this is a useful commitment — binds the reveal to a specific plaintext content. Without it, the operator could swap content post-mint (encrypt different bytes with the same key). With it, the dashboard verifies `sha256(decrypted plaintext) === on-chain plaintextHash`. Recommendation: skip for v0.3.0; sha256 inside the encrypted JSON is already part of each entry's signature, so tampering with content invalidates the per-entry signatures. We get this property for free.

## 10. Submission narrative (what we tell judges)

> **Verifiable Agent Execution — private receipts on 0G.**
>
> When your AI agent does something — runs a tool, makes a decision, executes an action — it leaves a cryptographic receipt. The receipt is signed by the agent's wallet, hashed, anchored as an iNFT (ERC-7857 AgenticID) on 0G Chain, and stored encrypted on 0G Storage. The full content is private by default. You own the key.
>
> Want to prove to anyone — investor, auditor, counterparty, partner — that your agent really did what it claimed? Generate a one-click share URL: `https://verifiable.0g.ai/verify/<tokenId>#k=<key>`. They see the proof chain flip green in their browser (your agent's wallet signed it, the storage hash matches, the iNFT anchor is valid). Without the key, all they see is metadata: "this attested action exists, owned by 0xAbu…".
>
> Same architecture as Phala / Ritual / Sentient for the privacy + verification layers. Same iNFT identity model as 0G's own AIverse / agentic-ID examples. Built on 0G's full stack: Storage (encrypted blobs), Chain (iNFT anchors), Compute (TEE-attested optional, demonstrated on tokenId 0). One npm-installable OpenClaw plugin — `openclaw plugins install @blockchainoracle/openclaw-verifiable-execution`.
