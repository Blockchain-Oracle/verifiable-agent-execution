# v0.3.4 — One Agent Task = One Token (revised after Codex round-4 design review)

**Status:** DRAFT v5 — Codex design-review-round-4 NEEDS-REVISION findings folded in

## Round-4 findings addressed in v5

1. **P1: `pendingAnchors` retains plaintext loggers too long.** v4 said registry survives ANY failure. But after `flush` succeeds, the encrypted bytes are on 0G Storage — recovery only needs `rootHash, entryCount, sessionId, dataDescriptionPrefix`, NOT the plaintext logger. Holding it past flush is a memory + privacy regression. **Narrow scope:** registry is ONLY for unflushed-logger recovery. Delete on `SessionAnchorMintAfterFlushError`, on post-mint commit failure, and on success. Pre-flush failures are the only path that keeps the logger.

2. **P2: Conflicting ordering in plan snippets.** Normative section (`plan:52`) says `setPending → takeAndRelease`. But code sketch (`plan:73`) and entry-flow diagram (`plan:155`) show `takeAndRelease` first. **Fix:** make all examples consistent — `setPending → takeAndRelease → pendingAnchors.set → flush`.

Codex round-4 confirmed OK:
- No durable-outbox required (matches current v0.3.0 in-memory pre-flush semantics)
- Same-process concurrency safe if `setPending`+`takeAndRelease` are synchronous + adjacent before the first await
- Subagent edge OK (distinct child sessionKeys)
- `buildPluginStateForTests` helper at `skill.test.ts:535` needs `pendingAnchors` added — noted

## Round-3 findings addressed in v4

1. **P1: post-rotate pre-flush loss path** — v3 only covered `setPending` failure. If `flush` fails AFTER `takeAndRelease`, entries die on the stack. **Fix:** add `state.pendingAnchors: Map<pendingKeyName, SessionLogger>`. The captured logger is registered there before flush; removed on success; stays for retry on failure. Keyed by `pendingKeyName` (not sessionKey) so it CANNOT race with the next turn on the same sessionKey.
2. **P2: `dataDescriptionPrefix` must flow into `retryMint()`** — both `anchor()` and `retryMint()` share `mintAndBuildResult` (`SessionAnchor.ts:287`). Propagate the prefix through the post-flush error so a manual `retryMint({rootHash, entryCount, sessionId, dataDescriptionPrefix})` can preserve `exec-log-orphan:`.
3. **P2: `parseSessionIdFromDescription` at `verify-proof.ts:505` also hardcodes `exec-log:`** — added to the centralization list. Helper replaces ALL 5 callsites now (was: 4).

Codex round-3 confirmed OK:
- Keystore metadata decoupling backwards-compat (old sidecars without `runId` still parse)
- `randomBytes(16)` resolves entropy
- `runId` in success log NOT a privacy regression (it's a random UUID, not routing-bearing like `sessionKey`)
- v0.3.3 full-content capture preserved in all four handlers
**Date:** 2026-05-14
**Trigger:** Abu's 2026-05-13 feedback on token 65: receipts split across multiple tokens, content too thin.

## Round-2 findings addressed in v3

1. **Retry semantics on pre-flush failure** — `takeAndRelease` MOVED to AFTER `setPending` succeeds (was: before). If `setPending` fails, logger stays in `SessionManager` so the operator can retry by reissuing the agent run.
2. **SessionAnchor `dataDescription` override** — added file `packages/chain-client/src/SessionAnchor.ts` to the change list. `AnchorInput` gets an optional `dataDescriptionPrefix?: string` (default `"exec-log"`).
3. **Centralize `exec-log[-orphan]:` parsing** — new helper `apps/dashboard/src/lib/exec-log-parser.ts` with `isExecutionLogDescription()` + `parseExecutionLogDescription()`. Fix all 4 callsites: `feed.ts:307`, `verify-proof.ts:290`, `:363`, `:649`.
4. **Decouple `pendingKeyName` from `sessionKey`** — `keystore.commitPending(pendingKeyName, tokenId, metadata: {sessionKey, runId})`: filesystem identity is the compound `${sessionKey}|run:${runId}`; user-visible `last-receipt.json` stores the ORIGINAL `sessionKey` + `runId` separately. `listPending()` returns `{sessionKey, runId, sanitizedFilename, createdAt}`. `/share` footnote shows the bare `sessionKey`, not the compound form.
5. **`randomBytes(16)`** — bumped from 4 bytes for the anon-runId fallback.

## The bug

Currently the plugin anchors a token on **both** `session_end` AND `agent_end`. `handleAgentEnd` is literally a one-line delegation to `handleSessionEnd` (`src/index.ts:1054-1059`). One user message → multiple tokens. Audit story fragments.

## Unit of audit

Per `/tmp/openclaw-src/src/plugins/hook-types.ts:277` + `:523`:

| Event | Semantics |
|---|---|
| `agent_end` | One agent run completes (`runId`, `messages[]`, `success`). **One user message → one agent response.** |
| `session_end` | Channel-level session reset/idle/compaction/restart. Lifecycle, not task. |

**One agent_end == one token.** This is what the auditor wants.

## Architectural change

### 1. SessionManager — atomic rotate (Codex round-1 finding #1: race)

**Problem.** `agent_end` is fire-and-forget in OpenClaw (`/tmp/openclaw-src/src/agents/harness/lifecycle-hook-helpers.ts:96-101`). Our anchor body holds the SessionLogger for seconds (upload + mint + commit). During that window, a NEW `message_received` on the same `sessionKey` would currently call `SessionManager.getOrCreate(sessionKey)` and get back the SAME instance — but `SessionLogger.appendEntry()` rejects on a flushing/flushed logger (`packages/logger/src/SessionLogger.ts:120-128`). The plugin catches and logs the failure, so entries from the NEW turn get silently lost (`src/index.ts:535-540`).

**Fix.** Add `SessionManager.takeAndRelease(sessionKey): SessionLogger | null`. **Round-3 correction**: also add a `pendingAnchors: Map<pendingKeyName, SessionLogger>` to `PluginState` so a logger removed from the active sessionKey map but failing mid-anchor is still retrievable for retry. Sequence in `handleAgentEnd`:

1. Look up logger via `state.sessions.has(sessionKey)`; bail if no entries.
2. Generate key K + run identity (real `runId` from event, or `anon-${randomBytes(16).toString("hex")}` fallback). Build `pendingKeyName = \`${sessionKey}|run:${runId}\``.
3. `setPending(pendingKeyName, K, {sessionKey, runId})` — synchronous file write. If FAILS, log ERROR + return; **logger stays in SessionManager** so a follow-up agent_end can retry on the same sessionKey.
4. `takeAndRelease(sessionKey)` — synchronous map move. New `message_received` on the same sessionKey gets a fresh logger.
5. `state.pendingAnchors.set(pendingKeyName, logger)` — synchronous. Registry holds the UN-flushed logger ONLY. Keyed by compound `pendingKeyName`.
   **(Steps 3 → 4 → 5 are all synchronous and run before any `await`. JS event-loop guarantee: no other hook can interleave.)**
6. `flush({encrypt})` — async. If FAILS, log ERROR with `pendingKeyName`; logger STAYS in registry for retry. **This is the ONLY failure path that keeps the registry entry.**
7. `mint(rootHash, {dataDescriptionPrefix})` — async. The flush has already succeeded → bytes are on 0G Storage. The plaintext logger is no longer needed for recovery (rootHash + entryCount + sessionId + dataDescriptionPrefix are enough). **`state.pendingAnchors.delete(pendingKeyName)` here** to free memory. If mint FAILS, log ERROR with concrete retryMint hint embedding rootHash + dataDescriptionPrefix; the keystore pending key remains for manual binding.
8. `commitPending(pendingKeyName, tokenId, {sessionKey, runId})` — on success: done. On failure: log ERROR; mint already succeeded, operator finishes binding via the structured-log hint. Registry already cleared in step 7.

The retry registry is **strictly bounded to pre-flush state** — never holds plaintext loggers after the encrypted bytes are durable. In practice ≤ 1 entry per active session, and only during the upload window (sub-second on Galileo). Operators inspect it via `state.pendingAnchors.keys()` for debug; the keystore's on-disk `listPending()` is the durable shadow of step 3.

```ts
// SessionManager addition
takeAndRelease(sessionKey: string): SessionLogger | null {
  const existing = this.map.get(sessionKey);
  if (existing === undefined) return null;
  this.map.delete(sessionKey);
  return existing;
}
```

`handleAgentEnd` becomes (consistent ordering — round-4/5 fix):
```ts
if (!state.sessions.has(sessionKey)) return; // no entries accumulated; nothing to anchor
const encryptionKey = generateKey();
const runId = event.runId ?? `anon-${randomBytes(16).toString("hex")}`;
const pendingKeyName = `${sessionKey}|run:${runId}`;

// 1. setPending (synchronous; if it throws, logger stays in SessionManager for retry)
state.keystore.setPending(pendingKeyName, encryptionKey, { sessionKey, runId });

// 2. takeAndRelease (synchronous; new message_received gets a fresh logger)
const logger = state.sessions.takeAndRelease(sessionKey);
if (logger === null) return; // race-impossible but defensive

// 3. pendingAnchors.set (synchronous; registry holds UN-flushed logger only)
state.pendingAnchors.set(pendingKeyName, logger);

// Steps 1-3 are all synchronous, no `await` between them — JS event-loop
// guarantee: no other hook can interleave.

// 4. flush + mint (async; one try/catch). `result` is HOISTED with
//    `let` so the post-mint commit block can reference txHash/
//    rootHash/entryCount (round-7 fix).
let result: { tokenId: bigint; txHash: string; rootHash: string; entryCount: number; verifyUrl: string };
try {
  result = await anchor.anchor({ /* ..., dataDescriptionPrefix: "exec-log", */ encrypt: encryptCb });
  // 5. mint succeeded → bytes durable on 0G Storage. Free the
  //    plaintext logger NOW. (Whether or not commitPending succeeds
  //    next, the registry should NOT hold the plaintext past mint.)
  state.pendingAnchors.delete(pendingKeyName);
} catch (cause) {
  if (cause instanceof SessionAnchorMintAfterFlushError) {
    // Bytes durable; plaintext no longer needed for recovery.
    state.pendingAnchors.delete(pendingKeyName);
    structuredLog("ERROR", "agent_end", "Mint failed after successful flush", {
      runId, rootHash: cause.rootHash, entryCount: cause.entryCount,
      sessionId: cause.sessionId, dataDescriptionPrefix: cause.dataDescriptionPrefix,
      recovery: `Call anchor.retryMint({rootHash:"${cause.rootHash}", entryCount:${cause.entryCount}, sessionId:"${cause.sessionId}", dataDescriptionPrefix:"${cause.dataDescriptionPrefix}"}) → captures <newTokenId>. Then keystore.commitPending("${pendingKeyName}", <newTokenId>, {sessionKey:"${sessionKey}", runId:"${runId}"}).`,
    });
  } else {
    // Pre-flush failure → registry keeps logger for retry.
    structuredLog("ERROR", "agent_end", "Flush failed before mint", {
      runId, pendingKeyName, sessionKey,
      recovery:
        `Logger retained in state.pendingAnchors["${pendingKeyName}"]. ` +
        `Recovery steps: ` +
        `(1) const logger = state.pendingAnchors.get("${pendingKeyName}"); ` +
        `(2) await logger.flush({encrypt}) → captures <rootHash>; ` +
        `(3) await agenticIdClient.mint(agentId, [{dataDescription:"exec-log:<sessionId>:<modelId>", dataHash:<rootHash>}]) → captures <newTokenId>; ` +
        `(4) state.keystore.commitPending("${pendingKeyName}", <newTokenId>, {sessionKey:"${sessionKey}", runId:"${runId}"}); ` +
        `(5) state.pendingAnchors.delete("${pendingKeyName}").`,
    });
  }
  return;
}

// 6. commitPending — SEPARATE try/catch (round-6 finding).
//    By this point mint succeeded, registry is already cleared.
//    A commit failure means the AES key on disk isn't yet bound to
//    the minted tokenId. Operator finishes binding manually using
//    the durable hint below.
const tokenId = result.tokenId.toString();
try {
  state.keystore.commitPending(pendingKeyName, tokenId, { sessionKey, runId });
  structuredLog("INFO", "agent_end", "Session anchored on-chain", {
    runId, tokenId, txHash: result.txHash, rootHash: result.rootHash,
    entryCount: result.entryCount, verifyUrl: result.verifyUrl,
    // NB: shareUrl + sessionKey intentionally omitted (round-9 + round-16)
  });
} catch (commitErr) {
  structuredLog("ERROR", "agent_end", "Keystore commit failed AFTER successful mint — receipt is on-chain but local key isn't yet bound to tokenId", {
    runId, tokenId, txHash: result.txHash, rootHash: result.rootHash,
    pendingKeyName, sessionKey,
    cause: commitErr instanceof Error ? commitErr.message : String(commitErr),
    recovery: `Receipt minted as tokenId ${tokenId} (tx ${result.txHash}). To bind the local key for /share, fix the FS issue then run: keystore.commitPending("${pendingKeyName}", "${tokenId}", {sessionKey:"${sessionKey}", runId:"${runId}"}). If the pending key file is gone, the receipt's contents are unrecoverable from THIS host.`,
  });
}
```

### 2. Pending key — run-scoped identity (Codex round-1 finding #2: collision)

**Problem.** If two `agent_end`s overlap on the same `sessionKey` (rare but possible: user sends two messages, harness queues two runs), both call `setPending(sessionKey, K)` → second overwrites first → token T1 commits with K2 → `/share T1` emits K2 → cross-token decryption. Same severity class as round-14.

**Fix.** Pending key identity is `${sessionKey}|run:${runId}`. The base64url encoder (round-14) keeps these collision-free. Falls back to a synthetic id if `event.runId` is missing.

```ts
const runId =
  typeof event.runId === "string" && event.runId.length > 0
    ? event.runId
    : `anon-${randomBytes(16).toString("hex")}`;
const pendingKeyName = `${sessionKey}|run:${runId}`;
state.keystore.setPending(pendingKeyName, encryptionKey, { sessionKey, runId });
// ... after mint
state.keystore.commitPending(pendingKeyName, tokenId, { sessionKey, runId });
```

**Keystore API change (Codex round-2 finding #4):** `setPending` and
`commitPending` take an optional metadata bag `{ sessionKey, runId }`.
The filesystem identity (`<base64url(pendingKeyName)>.key`) and the
sidecar metadata + last-receipt pointer are decoupled. `listPending()`
returns `{ sessionKey, runId, sanitizedFilename, createdAt }` so
operators see the ORIGINAL sessionKey, never the compound form.
`/share` no-args footnote renders the bare sessionKey.

### 3. session_end — orphan recovery (Codex round-1 finding "session_end contradiction")

**Problem.** If `session_end` is pure no-op, runs whose harness crashed mid-flight (no `agent_end`) lose entries. If `session_end` always mints, the "one agent_end = one token" promise breaks.

**Fix.** `session_end` checks if any orphaned SessionLogger entries exist for this `sessionKey` (via `state.sessions.has(sessionKey)`). If yes, MINT an ORPHAN token with distinct `dataDescription` prefix `exec-log-orphan:<sessionId>:<modelId>` and an ERROR-level structured log explaining "session ended without agent_end; recovery anchor minted." If no, no-op.

```ts
const orphan = state.sessions.takeAndRelease(sessionKey);
if (orphan === null) return; // normal path: agent_end already cleared the logger
// abnormal path: harness died mid-run; anchor what we have so the bytes
// aren't lost (caller asked for it)
structuredLog("ERROR", "session_end", "Orphan recovery anchor — agent_end never fired", {
  sessionId, /* sessionKey omitted per round-16 privacy */
});
// anchor using dataDescription "exec-log-orphan:..." so the dashboard
// renders the orphan-recovery badge
```

Dashboard distinguishes orphan tokens via the `dataDescription` prefix.

### 4. Heartbeat detection — DROPPED in v0.3.4 (Codex round-1 finding #3: unsafe heuristic)

**Problem.** My proposed rule "entryCount === 2 && no message_received && no after_tool_call" false-positives on legitimate single-LLM-response runs — `llm-task` extension uses `disableTools: true` (`/tmp/openclaw-src/extensions/llm-task/src/llm-task-tool.ts:261-276`), `voice-call` does single embedded responses (`/tmp/openclaw-src/extensions/voice-call/src/response-generator.ts:315-330`).

**Fix.** v0.3.4 anchors **every** `agent_end`. Heartbeat-detection is deferred to v0.4.0 where it needs an explicit signal (e.g., a runtime-provided `event.trigger === "heartbeat"` flag, OR a known heartbeat sessionKey lane). We don't infer from entry shape.

Feed-spam mitigation lives elsewhere (dashboard filtering, future runtime "trigger" metadata).

### 5. Subagent semantics — corrected (Codex round-1 finding #4)

Per Codex review, two distinct subagent shapes exist:

- **MCP-embedded subagent** (e.g., `active-memory` at `/tmp/openclaw-src/extensions/active-memory/index.ts:2459-2475`, Google Meet at `/tmp/openclaw-src/extensions/google-meet/src/agent-consult.ts:55-68`) — uses a CHILD `sessionKey`. Fires its OWN `agent_end`. ⇒ **Gets its own token** (correct under "one agent_end = one token").

- **Codex native subagent** (`/tmp/openclaw-src/extensions/codex/src/app-server/event-projector.ts:148-161`, `native-subagent-task-mirror.ts:248-273`) — mirrored as task lifecycle inside the parent's Codex app-server projector. Does NOT fire a separate `agent_end`. ⇒ **Folded into the parent's token.** Parent's entries include the child's tool calls as nested `after_tool_call` events.

Either shape is handled correctly by "anchor on agent_end" — no special-casing needed.

## Entry flow (revised)

```
sessionKey: agent:research:telegram:group:-1003.../topic:5
runId:      993accba-f307-458b-ab4c-65d337737ea5

─── turn N ───────────────────────────────────────────────────
   #000  message_received   "ETH price + 0G news?"
   #001  before_prompt_build prompt+system+history (full v0.3.3)
   #002  after_tool_call    web_search { query: "ETH price"}
   #003  after_tool_call    web_search { query: "0G news"}
   #004  llm_output         "Here are 3 bullets: ..."
   #005  agent_end          { runId, success, durationMs }
─── handler ─────────────────────────────────────────────────
   // ALL SYNCHRONOUS — no await between these steps:
   keystore.setPending(`${sessionKey}|run:${runId}`, K, {sessionKey, runId})
   logger = state.sessions.takeAndRelease(sessionKey)
   state.pendingAnchors.set(pendingKeyName, logger)
   // Now async — registry holds unflushed logger:
   logger.flush({encrypt}) → 0G Storage rootHash
   AgenticID.mint(rootHash, {dataDescriptionPrefix:"exec-log"}) → tokenId
   state.pendingAnchors.delete(pendingKeyName)  // bytes are durable; free memory
   keystore.commitPending(`${sessionKey}|run:${runId}`, tokenId, {sessionKey, runId})
   structuredLog INFO "Session anchored on-chain" {tokenId, txHash, rootHash, entryCount, verifyUrl, runId}
─── turn N+1 starts ─────────────────────────────────────────
   message_received → SessionManager.getOrCreate(sessionKey) → FRESH logger
   ... new entries accumulate in new logger ...
```
