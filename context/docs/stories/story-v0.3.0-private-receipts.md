## Narrative

As the operator of a verifiable AI agent, when my agent finishes a
session I want the receipt's payload encrypted on 0G Storage so that
public verification still works (proof chain stays auditable) but the
**content** stays private. I then want to deliberately share a viewer
URL with the `#k=<key>` fragment when — and only when — I choose to.

This kills the "Etherscan for AI agents" privacy violation Abu flagged
2026-05-08: by default, agent runs leaked tool params/results to anyone
who scraped the AgenticID feed. v0.3.0 makes encryption the default
and selective reveal an explicit operator action via `/share`.

The wedge stays: TEE-rooted verifiable receipts. The user surface
matches Phala dstack, EAS PrivateData, Ritual, Sentient — encrypt by
default; reveal on consent.

## Acceptance criteria

```gherkin
# --- m1: AES-256-GCM crypto foundation ---
Given a 32-byte key generated by generateKey()
When encryptSessionLog(plaintextJson, key) is called
Then it returns a v1 envelope {v: 1, alg: "aes-256-gcm", iv, ciphertext, tag}
And the IV is 12 bytes random per call (never reused)
And the authTag is exactly 16 bytes (CWE-310 defense)
# alg uses the lowercase Node-crypto identifier on-wire (matches the
# `createCipheriv("aes-256-gcm", ...)` call that produced the bytes).
# The dashboard's WebCrypto decrypter normalizes to "AES-GCM" internally.

Given an encrypted envelope and the matching key
When decryptSessionLog(envelope, key) is called
Then it returns the original plaintext bytes
And tampering with any of {iv, ciphertext, tag} causes decryption to throw

Given an envelope with v != 1
When decryptSessionLog(envelope, key) is called
Then it throws "Unsupported envelope version" (forward-compat trap)

# --- m2: Keystore with crash-recovery ---
Given a fresh Keystore at ~/.openclaw/verifiable-execution/keystore
When put(tokenId, key) is called
Then a file at keystore/<tokenId>.key exists with mode 0600
And last-receipt.json is updated to point at this tokenId

Given setPending(sessionKey, key) is called before mint succeeds
And the process crashes between flush and mint
When the process restarts and inspects keystore/pending/
Then the pending key file survives, so a retryMint can still anchor it

Given a sessionKey containing OpenClaw separators ("agent:core:telegram:direct:8028...")
When setPending(sessionKey, key) is called
Then the sanitized filename has no ":" or "/" (filesystem-safe)
And the original sessionKey is recoverable by callers via listPending()
# (NOT list() — list() returns committed tokenIds only. Pending
# crash-recovery surfaces through listPending(), which reads sidecar
# .meta.json files alongside each pending .key. Codex round-3 surfaced
# the original BDD wording inconsistency; resolved by renaming the API
# in the spec to match the implementation that ships in v0.3.0.)

# --- m3: Encrypt-before-flush ordering ---
Given SessionLogger.flush() receives FlushOptions {encrypt}
When the session ends and the plugin calls anchor.anchor({encrypt})
Then the order is:
  1. generateKey(K) for this session
  2. keystore.setPending(sessionKey, K)
  3. SessionLogger.flush() invokes encrypt(plaintextJson) before upload
  4. 0G Storage receives an encrypted envelope blob, NOT plaintext
  5. AgenticID.mint anchors the rootHash of the encrypted blob
  6. keystore.commitPending(sessionKey, tokenId) — moves to keystore/<tokenId>.key
  7. structuredLog emits a KEY-FREE "Session anchored on-chain" line with
     {tokenId, txHash, rootHash, entryCount, verifyUrl} — the reveal key
     and `#k=` fragment are intentionally OMITTED from logs (Codex
     round-9 SECURITY fix: auto-logging the key on every session leaked
     decryption material to gateway / collector / observability logs,
     defeating the encrypted-by-default contract). Operators fetch the
     full share URL on demand via the `/share` command, which reads the
     committed key from the keystore at request time.

Given flush succeeds but mint fails
When the operator inspects keystore/pending/
Then the key is still there for retryMint
And the structuredLog ERROR includes a copy-paste retryMint recovery hint

# --- m4: Dashboard dual-mode render (server is KEY-BLIND by design) ---
# Codex round-1 surfaced that the earlier "server decrypts with ?k="
# design leaked the reveal key into HTTP access logs. Resolution: the
# server is intentionally key-blind. All decryption happens client-side
# in the browser via WebCrypto; the server only serves metadata + the
# raw encrypted envelope passthrough.

Given /api/verify/<tokenId> resolves an encrypted blob
Then the API returns ProofResponse {verified: "encrypted", entries omitted, metadata: present}
And meta.rpcUrl + meta.verifierAddress are populated so the client
  can run on-chain verifyTEESignature in the browser

Given /api/verify/<tokenId>/blob is hit (a key-blind passthrough)
Then the API returns the raw EncryptedSessionLogEnvelope JSON
  (no decryption, no key parsing, no key in the URL)

Given /verify/<tokenId> is loaded server-side with proof.verified === "encrypted"
Then page.tsx renders <EncryptedReveal initialProof={proof} /> (client component)
And EncryptedReveal reads window.location.hash for "#k="
And fetches /api/verify/<tokenId>/blob (NO ?k= in the URL)
And decrypts the envelope client-side via WebCrypto AES-GCM
And synthesizes the full ProofResponse with decoded entries
And renders SessionView with a client-side verifyEntry callback
  (per-entry verify uses browser-side ethers — ecrecover + on-chain
  MockTEEVerifier.verifyTEESignature via the public 0G RPC)

Given an attacker appends ?k=<key> to /api/verify/<tokenId>
When the route handler processes the request
Then the response is IDENTICAL to the no-?k= request (key-blind by signature)
And the key NEVER appears in server logs / reverse proxy logs / APM traces

# --- m5: /share slash command UX ---
Given the operator types "/share" in the agent's chat (Telegram, CLI)
And the channel passes `content: "/share"` (or "/share <id>") AND `commandAuthorized: true`
When the inbound_claim event fires
Then handleShareCommand(ctx, event) returns {handled: true, reply: {text: ...}}
And the reply contains a URL of the form
  "https://verifiable.0g.ai/verify/<lastTokenId>#k=<base64url>"
And the reply mentions the session key for the operator's own debugging
# v0.3.0 explicitly REQUIRES `content: "/share"` because OpenClaw 2026.4.x
# does not surface a `command`/`commandName` field on PluginHookInboundClaimEvent
# (see /tmp/openclaw-src/src/plugins/hook-message.types.ts:26). Without that
# signal, plugins must self-discriminate by content text. Channels that
# pre-split slash-command args without raw content (Discord-style) are
# intentionally rejected — see Out of scope below. The fix requires
# wiring ctx.pluginBinding like the stock codex plugin, which is v0.4.0
# scope.

Given the operator types "/share <specificTokenId>"
Then the reply targets that tokenId, NOT the last receipt
And no session footnote is included (privacy: don't leak unrelated sessionKey)

Given the keystore has no receipts yet
When "/share" is typed
Then the reply is "📭 No receipts yet…" with no URL or key string emitted

Given the keystore has no key for the requested tokenId
(e.g., minted on a different host or wiped)
When "/share <tokenId>" is typed
Then the reply is "🔒 No key on this host for tokenId <id>…" with no #k= leak

Given verifyUrlBase = "https://verifiable.0g.ai/" (trailing slash)
When the share URL is built
Then there's no "//" before "/verify"

# --- Cross-cutting: backward compatibility ---
Given a v0.2.0 legacy plaintext receipt (e.g., tokenId 0)
When /verify/0 is loaded
Then the page renders SessionView directly (not EncryptedReveal)
And StatusBadge shows "verified" | "preview" | "unverified" — NOT "encrypted"

Given the §14 grep gate runs on hot-path source
When src/, lib/, packages/*/src/ are scanned
Then no token in {mock, fake, dummy, hardcoded} is matched
And contracts/contracts/MockTEEVerifier.sol is exempt per ADR-06
```

## Out of scope (deferred)

- **EAS PrivateData migration** — discussed; deferred to v0.4.0 (uses 0G's iNFT model for v0.3.0 to stay native to the sponsor stack)
- **CLI subcommand for share** — included as `/share` slash command; standalone `openclaw share <tokenId>` CLI deferred (no operator demand yet)
- **before_message_write auto-inject** — REJECTED per Abu's UX critique 2026-05-12 ("99% of the time you never share — polluting the conversation with a verifier link is friction, not UX")
- **plaintextHash** — skipped; the encrypted blob's rootHash is the only on-chain commitment, which is sufficient for the proof chain
- **Trustless TEE-attestation-only "verified" badge** — v0.3.0 inherits the v0.2.0 trust model: ecrecover-match against `entry.agentId` is sufficient for the "verified" badge, falling back to MockTEEVerifier (oracle-signed) only when ecrecover doesn't match. Codex round-5 flagged this as "anyone can self-sign green" — true, and ADR-10 explicitly frames the wedge as "TEE-rooted, **not trustless**." A trustless model requires real Phala dstack / 0G TEE oracle integration where the agentId itself is bound to a remote attestation quote — v0.4.0 scope. The current pinning test lives in `apps/dashboard/tests/verifier-route.test.ts` ("returns 'verified' when ecrecover matches agentId — even if MockTEEVerifier would reject") so accidental "fixes" can't regress the agent-wrapper attestation path.

## Spec evolution

The original v0.3.0 plan called for **auto-injecting** the share URL into every agent reply via `before_message_write`. Abu pushed back during implementation (2026-05-12): "you're not really thinking from a developer's perspective. You're just thinking to get work done." The replacement is the `/share` slash command — silent by default, explicit on demand. This matches the stock OpenClaw `codex` plugin's `registerCommand` + `inbound_claim` pattern.

The keystore originally lived in-memory. After researching crash-recovery requirements (mint succeeds but process dies before key is persisted = unrecoverable receipt), it was promoted to a file-backed `~/.openclaw/verifiable-execution/keystore/` store with mode 0600 and a `pending/` subdir that holds keys between flush and mint.
